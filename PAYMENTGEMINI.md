# HarusiYangu Financial Integrity & Payment Mandates (PAYMENTGEMINI.md)

This document defines the strict engineering standards for the Financial, Wallet, and Settlement systems within the HarusiYangu platform. Every financial operation must adhere to these rules to ensure "Real Money" integrity.

## 1. The Principle of Atomic Integrity (No Lost Money)
Every movement of funds (Contribution -> Event Wallet -> Vendor Wallet -> Payout) must be wrapped in a **Database Transaction**.
- **Mandate:** Use `DB::beginTransaction()`, `DB::commit()`, and `DB::rollBack()`.
- **Constraint:** A financial record is NEVER valid unless its corresponding Ledger Entry and Notification are successfully created. If one fails, all fail.

## 2. Triple-Entry Traceability (The Immutable Ledger)
We do not simply update balances; we record history.
- **The Ledger:** Every change to `current_balance`, `pending_balance`, or `available_balance` must create a record in `wallet_ledger_entries`.
- **Source of Truth:** If the sum of Ledger Entries for a wallet does not equal the Wallet's balance, the system must trigger a `FinancialInconsistencyException` and freeze the wallet.
- **Reference IDs:** Every entry must capture:
    - `internal_receipt_number`: Generated by HarusiYangu (e.g., HY-PAY-2026-X).
    - `transaction_reference`: The REAL reference from the external provider (M-Pesa, Bank, etc.).
    - `metadata`: The raw JSON response from the payment gateway to serve as forensic evidence.

## 3. The 30/30/40 Trust Protocol (Escrow Implementation)
To protect both Host and Vendor, the system acts as a trusted Escrow agent.
- **Milestone 1 (Deposit - 30%):** Direct Transfer. Moves from `EventWallet` to Vendor `available_balance`. Triggers `ACTIVE` event status.
- **Milestone 2 (Interim - 30%):** Direct Transfer. Moves from `EventWallet` to Vendor `available_balance`.
- **Milestone 3 (Final - 40%):** **Escrow Hold.** Moves from `EventWallet` to Vendor `pending_balance`.
- **Release Trigger:** The 40% only moves from `pending` to `available` when the Host executes the `confirmServiceReceived` action.

## 4. Real-World Settlement (Withdrawals & Payouts)
A balance in the system is a liability until it is settled to the Vendor's real bank/mobile account.

### 4.1 Vendor Settlement Accounts
Vendors must register their legal payout destinations.
- **Mobile Money:** Phone Number + Provider (M-Pesa, Airtel Money, Tigo Pesa).
- **Bank Accounts:** Bank Name + Account Number + Branch Code + Account Holder Name.
- **Verification Gate:** 
    - **Automated:** System verifies Mobile Money names against provider APIs where possible.
    - **Manual:** Vendors must upload a Bank Statement header or Cancelled Check for Super Admin verification of bank details.

### 4.2 Withdrawal Workflow
1. **Initiation:** Vendor selects an "Available Balance" amount and a verified Settlement Account.
2. **Security Lock:** Once a withdrawal is requested, the settlement account details are frozen for that transaction. Payout account changes trigger a 24-hour withdrawal cooldown.
3. **Execution:**
    - **Integrated (M-Pesa/API):** The system calls the Payout API (B2C) and waits for a Success Webhook.
    - **Non-Integrated (Banks/Other):** The system generates a "Pending Settlement" task for the Super Admin. Once the transfer is done manually, the Admin MUST upload the "Proof of Payment" (Reference ID/Scan) to the system to finalize the withdrawal.

## 5. Handling Non-Integrated Providers (The Manual Bridge)
Not every bank or telco has a perfect API. We handle this with professional "Manual Bridging":
- **Proof of Payment (PoP) Requirement:** For every manual bank or cash transfer, the system requires a `transaction_reference` and an **attachment upload** (Scan/Photo of Receipt).
- **Vendor Receipt Confirmation:** For manual transfers, the Vendor must click "Confirm Receipt" in their dashboard. This provides the Host with 100% certainty that the money has arrived.
- **Reconciliation Audit:** Manual payments remain in `UNVERIFIED` status until both the Vendor confirms receipt and a Super Admin cross-references the bank statement to mark it as `VERIFIED`.

## 6. Automated Reconciliation (The Auditor)
The system must be self-correcting.
- **Reconciliation Job:** A scheduled task that cross-references `provider_reference` IDs with integrated APIs (e.g., Vodacom M-Pesa) every hour.
- **Forensic Logging:** If an automated API fails or returns an error, the system creates a "High Priority Financial Alert" for the Admin.

## 7. Financial Reporting & Legal Documents
For every transaction, the system must generate:
- **Digital Receipt (PDF):** For the Host (Proof of Payment).
- **Remittance Advice (PDF):** For the Vendor (Proof of Earnings/Withdrawal).
- **Audit Log:** Viewable by Super Admin for dispute resolution.

---
**Core Rule:** In HarusiYangu, code handles the logic, but the Ledger handles the truth. Never update a balance without a receipt.
